## Analysis & Reasoning
The PR introduces Prisma ORM for the first time — adding `prisma/schema.prisma`, migrations, `PrismaService`, and the `@prisma/client`/`prisma` packages. However, the generated Prisma client was never created.

`@prisma/client` ships only a thin stub that re-exports from `node_modules/.prisma/client`, a directory that is populated exclusively by running `prisma generate`. Without that step, `PrismaClient` is not exported, and no model-typed properties (`workspace`, `document`, `annotation`, `operationalRule`, `feedbackEvent`, `$transaction`, etc.) exist on the class — causing 37 TypeScript errors across every service file that touches Prisma.

The fix adds `"postinstall": "prisma generate"` to the `scripts` section in the root `package.json`.

This is the standard Prisma-recommended approach for monorepos and CI: whenever `npm install` runs (triggered in CI by the changed `package.json`), the `postinstall` hook fires immediately after, generating the typed client from `prisma/schema.prisma` into `node_modules/.prisma/client` before any build or test tasks execute.

No changes to individual service files were needed — they were all correct; the only missing piece was the generation step itself.
